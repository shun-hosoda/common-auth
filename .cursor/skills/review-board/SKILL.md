# Review Board — 5人専門家レビュー会議

## 概要

5人の専門家が**議論形式**でコードレビューを行う。
各自が独立に指摘するのではなく、互いの意見に反応・補足・反論し、合議で結論を出す。

## 事前準備

1. `docs/review/persona.md` を読み、ドメインペルソナを確認する
2. ペルソナが設定されていればそれを上乗せ、未設定ならデフォルトで進める

## レビュー対象の取得（コンテキスト最適化）

レビュー対象は以下の優先順位で取得し、**必要最小限の差分**をレビューする。

### 優先順位1: PRが存在する場合
```bash
gh pr diff
gh pr view --comments
```
- PR差分のみをレビュー対象とする
- 既存のPRコメントがあれば考慮する

### 優先順位2: ステージング済みの変更がある場合
```bash
git diff --cached
```

### 優先順位3: コミット間の差分
```bash
git diff HEAD~1
git log --oneline -5
```

### 優先順位4: 指定ファイルのレビュー
- ユーザーが指定したファイルを直接レビューする

## レビュー会議の進行

### Phase 1: 各専門家の初見（30秒レビュー）

5人がそれぞれの専門領域から**最も重要な1点**を挙げる。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  REVIEW BOARD — 初見ラウンド
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PM:       （ユーザー価値・要件整合の観点で1点）
Arch:     （構造・設計の観点で1点）
Eng:      （コード品質・保守性の観点で1点）
Sec:      （セキュリティの観点で1点）
DB:       （データ整合性・性能の観点で1点）
```

### Phase 2: 議論（クロスレビュー）

初見で挙がった指摘に対して、他の専門家が**反応**する。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  REVIEW BOARD — 議論
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Arch: 「Engの指摘に同意。さらに〇〇の観点からも問題がある」
Sec:  「その修正をする場合、△△のセキュリティリスクに注意」
Eng:  「Secの指摘を踏まえると、実装は□□のアプローチが良い」
DB:   「□□のアプローチだとクエリ性能が懸念。代案として...」
PM:   「その代案だとユーザー体験に影響する可能性がある」
```

### Phase 3: 判定

議論を踏まえ、最終判定を下す。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  REVIEW BOARD — 判定
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

判定: APPROVE / REQUEST_CHANGES / NEEDS_DISCUSSION

[MUST FIX] マージ前に必ず修正（番号付きリスト）
  1. ...
  2. ...

[SHOULD FIX] 強く推奨する改善（番号付きリスト）
  1. ...

[CONSIDER] 検討事項（番号付きリスト）
  1. ...

[GOOD] 特に評価する点
  - ...
```

## 深刻度の定義

| レベル | 意味 | 例 |
|--------|------|-----|
| MUST FIX | マージブロッカー。修正しないとリリース不可 | セキュリティ脆弱性、データ損失の可能性 |
| SHOULD FIX | 品質基準未達。技術的負債になる | エラーハンドリング不足、テスト不足 |
| CONSIDER | より良い方法がある。次回以降でも可 | 命名改善、パフォーマンス最適化 |

## レビューログの記録

レビュー完了後、`docs/review/logs/` にログファイルを作成する。

### ファイル名の生成

```
YYYY-MM-DD_HHmmss_<context>.md
```

- context: ブランチ名、PR番号、またはファイル名（スラッシュは`-`に変換）
- 例: `2026-02-28_143025_feature-auth.md`

### ログファイルの構造

```markdown
# Review Log — <タイトル>

## メタデータ
- 日時: 2026-02-28 14:30:25
- 対象: feature/auth-jwt（または PR #42, src/api/user.ts）
- レビュアー: Review Board（5人合議制）
- ラウンド: 1

## レビュー対象

（差分のサマリー、変更ファイル一覧、追加/削除行数）

## Phase 1: 初見ラウンド

PM: （発言内容）
Arch: （発言内容）
Eng: （発言内容）
Sec: （発言内容）
DB: （発言内容）

## Phase 2: 議論

（5人の議論のやりとり）

## Phase 3: 判定

判定: REQUEST_CHANGES

[MUST FIX]
1. src/api/auth.ts:42 — SQLインジェクションの可能性
2. src/api/auth.ts:67 — 認証エラー時に機密情報を返している

[SHOULD FIX]
1. src/api/auth.ts:15 — 関数が50行を超えている。分割を推奨

[CONSIDER]
1. tests/auth.test.ts — 異常系テストの追加を検討

[GOOD]
- バリデーションロジックが明確に分離されている
```

### 修正記録の追記（/fix実行時）

同じログファイルの末尾に追記する。

```markdown
---

## 修正記録

修正日時: 2026-02-28 15:10:45
実施者: AI Agent

修正内容:
- [MUST FIX #1] ✅ パラメータ化クエリに変更（src/api/auth.ts:42）
- [MUST FIX #2] ✅ エラーレスポンスを汎用メッセージに変更（src/api/auth.ts:67）
- [SHOULD FIX #1] ✅ 関数を3つに分割（src/api/auth.ts:15-65）

未対応:
- なし
```

### 再レビュー記録の追記（/re-review実行時）

```markdown
---

## 再レビュー

再レビュー日時: 2026-02-28 15:20:10
ラウンド: 2

前回の指摘:
- [MUST FIX #1] ✅ 解決済み
- [MUST FIX #2] ✅ 解決済み
- [SHOULD FIX #1] ✅ 解決済み

新規指摘: なし

最終判定: APPROVE ✅
```
