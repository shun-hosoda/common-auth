# Dev Cycle — 実装・テスト・レビュー・修正の自律サイクル

## 概要

実装計画に基づいて以下のサイクルを自律的に繰り返し実行する:

```
実装 → テスト → 失敗分析・修正 → 再テスト → レビュー → 修正 → 完了
```

ユーザー介入なしで可能な限り自律的に進め、判断が必要な場合のみ報告する。

## サイクル全体像

```
┌─────────────────────────────────────┐
│  STEP 1: 実装                       │
│  - 実装計画に基づきコード作成        │
│  - テストコード作成（TDD Green）     │
└───────────┬─────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│  STEP 2: テスト実行                  │
│  - pytest tests/unit/ -v --tb=short │
│  - 結果を分析                        │
└───────────┬─────────────────────────┘
            ↓
      ┌─────┴─────┐
      │ 全て合格？  │
      └─────┬─────┘
      Yes   │   No
       ↓    │    ↓
       │  ┌─┴────────────────────────┐
       │  │ STEP 3: 失敗分析・修正    │
       │  │ - エラーパターン特定       │
       │  │ - 自動修正               │
       │  │ - 再テスト（最大3回）     │
       │  └──────────────────────────┘
       ↓
┌─────────────────────────────────────┐
│  STEP 4: セルフレビュー              │
│  - 5人専門家によるコードレビュー     │
│  - MUST FIX / SHOULD FIX 特定       │
└───────────┬─────────────────────────┘
            ↓
      ┌─────┴─────┐
      │  APPROVE?  │
      └─────┬─────┘
      Yes   │   No
       ↓    │    ↓
       │  ┌─┴────────────────────────┐
       │  │ STEP 5: レビュー修正      │
       │  │ - MUST FIX修正           │
       │  │ - SHOULD FIX修正         │
       │  │ - 再テスト               │
       │  └──────────────────────────┘
       ↓
┌─────────────────────────────────────┐
│  STEP 6: 完了報告                    │
│  - 実装サマリー                      │
│  - テスト結果                        │
│  - レビュー結果                      │
│  - 次のアクション                    │
└─────────────────────────────────────┘
```

## 実行プロトコル

### STEP 1: 実装

実装計画書（`docs/implementation/logs/`）を参照し、コードを実装する。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — STEP 1: 実装
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

対象: <コンポーネント名>
ファイル:
  - src/common_auth/middleware/rate_limit.py （新規）
  - tests/unit/test_rate_limit.py （新規）

状態: 実装中...
```

**ルール**:
- TDDの場合: テスト → 実装 → リファクタリング
- 型ヒント必須
- docstring必須（公開API）
- コーディング規約を遵守

### STEP 2: テスト実行

```bash
cd packages/backend-sdk
$env:PYTHONPATH="src"; python -m pytest tests/unit/ -v --tb=short --no-cov
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — STEP 2: テスト実行
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

結果: 24 passed, 3 failed
実行時間: 2.1s
```

### STEP 3: 失敗分析・修正（テスト失敗時のみ）

**エラーパターンと自動修正**:

| パターン | 検出 | 修正 |
|----------|------|------|
| AsyncMock設定ミス | `coroutine object` | `.json = MagicMock(return_value=...)` |
| モック未await | `was never awaited` | `AsyncMock`使用 |
| バリデーションエラー | `ValidationError` | テストデータ修正 |
| Import不足 | `ModuleNotFoundError` | `pip install` |
| 型不一致 | `TypeError` | 型変換追加 |
| assertionエラー | `AssertionError` | 実装 or テスト修正 |

**修正ルール**:
1. テストが正しい → 実装を修正
2. 実装が正しい → テストを修正
3. 判断できない → 設計書を参照して判断

**リトライ上限**: 3回
- 3回失敗 → ユーザーに報告しエスカレーション

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — STEP 3: 修正 (Round 1/3)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

失敗: 3件
修正:
  1. test_rate_limit_store → TTLCacheインポート追加
  2. test_middleware_429 → レスポンスステータスコード検証修正
  3. test_x_forwarded_for → monkeypatch使用に変更

再テスト実行中...
```

### STEP 4: セルフレビュー

テスト全合格後、5人専門家による簡易レビューを実施。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — STEP 4: セルフレビュー
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PM:   要件との整合性確認 ✅
Arch: 構造・パターン確認 ✅
Eng:  コード品質確認 ⚠️ [SHOULD FIX] 関数が長すぎる
Sec:  セキュリティ確認 ✅
DB:   データ層確認 ✅

判定: APPROVE ✅ (SHOULD FIXあり)
```

**レビュー基準**:
- **MUST FIX**: セキュリティ脆弱性、データ不整合、破壊的バグ → 修正必須
- **SHOULD FIX**: コード品質、パフォーマンス、保守性 → 修正推奨
- **CONSIDER**: 将来改善事項 → 記録のみ

### STEP 5: レビュー修正（REQUEST_CHANGESの場合のみ）

MUST FIX/SHOULD FIXを修正し、再テストを実行。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — STEP 5: レビュー修正
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

修正:
  [SHOULD FIX #1] ✅ dispatch関数をヘルパーに分割

再テスト: 全合格 ✅
```

### STEP 6: 完了報告

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DEV CYCLE — 完了
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

実装:
  ✅ rate_limit.py — RateLimitStore + InMemoryRateLimitStore + Middleware
  ✅ test_rate_limit.py — 12テスト

テスト: 36 passed, 0 failed
レビュー: APPROVE ✅

修正ラウンド: テスト修正1回 + レビュー修正1回
合計所要時間: 〜5分

次のアクション:
  - 次のStepの `/cycle` を実行
  - または `/push` でコミット
```

## バッチ実行モード

複数のStepを連続で実行する場合:

```
/cycle --steps 1,2,3
```

各Step完了後に次のStepに自動移行:
1. Step完了 → テスト全合格 → レビューAPPROVE → 次のStep
2. テスト失敗 → 自動修正 → 再テスト（最大3回）
3. レビューREQUEST_CHANGES → 自動修正 → 再レビュー

## エスカレーション条件

以下の場合はユーザーに報告して判断を仰ぐ:
- テスト修正3回失敗
- MUST FIXが3ラウンド未解決
- 設計書と実装の乖離を検出
- 破壊的変更が必要な場合
