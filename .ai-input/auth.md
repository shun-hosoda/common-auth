# ポータブル認証プラットフォーム 設計仕様書 (v1.0)

## 1. 目的と設計思想
本ドキュメントは、特定のインフラ（AWS/Azure/GCP）に依存せず、顧客環境ごとに独立してデプロイ可能な「共通ログイン認証モジュール」の設計指針を定義する。

### 基本原則
- **Don't Roll Your Own Crypto/Auth**: 認証ロジックを自前で実装せず、実績のあるOSS（Keycloak）を採用する。
- **Stateless & Scalable**: バックエンドはセッションを持たず、JWT（JSON Web Token）による検証のみを行う。
- **Portable Architecture**: インフラ固有のマネージドサービスを避け、Dockerコンテナによる「どこでも動く」構成を維持する。

---

## 2. システムアーキテクチャ
認証機能を提供する「Auth Stack」と、業務ロジックを持つ「App Stack」を完全に分離する。



### コンポーネント構成
1. **Identity Provider (IdP)**: Keycloak (Docker)
   - ユーザー管理、認証画面、MFA、パスワードリセットを担当。
2. **Frontend**: React + TypeScript
   - OIDCプロトコルに基づき、IdPからトークンを取得。
3. **Backend**: Python (FastAPI/Flask)
   - JWTの署名検証、セキュリティヘッダ付与、テナント識別を担当。
4. **Application DB**: PostgreSQL (or others)
   - 業務データのみを保持。ユーザーのパスワード等は保持しない。

---

## 3. 実装機能一覧

| カテゴリ | 機能項目 | 実装詳細 |
| :--- | :--- | :--- |
| **認証 (Auth)** | ログイン / ログアウト | OIDC Authorization Code Flow + PKCE |
| | 二段階認証 (MFA) | Keycloak標準 (Google Authenticator等) |
| | パスワード再設定 | セルフサービス (Email連携) |
| **管理** | ユーザーセルフ登録 | メルアド確認プロセスを含む |
| | マルチテナント | Realm (レルム) による論理隔離 |
| **セキュリティ** | 通信改ざん防止 | JWT署名検証 (RS256) |
| | 大量攻撃対策 | Rate Limit ミドルウェアによるIP制限 |
| | セキュリティヘッダ | HSTS, CSP, X-Frame-Options 等の強制付与 |

## 4. マルチテナント・データ分離設計
顧客ごとにインフラを構築する場合でも、データ整合性と隔離を担保する。

### データの紐付け方法
1. Keycloakが発行するJWT内の `sub` クレーム（不変のUUID）をユーザーの主キーとする。
2. 業務DB内の全テーブルに `tenant_id` カラムを定義。
3. PythonミドルウェアでJWTをデコードし、抽出した `tenant_id` でDBクエリを強制的にフィルタリングする。

---

## 5. 配布と運用のプロセス

### 5.1 顧客環境へのデプロイ
- **環境差異の吸収**: SMTPサーバー（メール送信）やDB接続情報は、すべて環境変数 (`.env`) で注入する。
- **設定のポータビリティ**: Keycloakの設定（Realm設定）をJSONファイルとして管理し、コンテナ起動時に自動読み込みを行う。

### 5.2 プロジェクト間での再利用方法
- **Frontend SDK**: ログイン処理をカプセル化した共通Hooksを作成。
- **Backend SDK**: JWT検証とセキュリティヘッダ付与を行う共通ミドルウェアをパッケージ化（PyPI等で管理）。

---

## 6. 将来の拡張性
- **ソーシャルログイン**: 設定変更のみでGoogleやMicrosoft Entra ID連携が可能。
- **クラウドマネージド移行**: 規模拡大時に、アプリコードを変更せず AWS Cognito 等へ移行可能（OIDC規格に準拠しているため）。
