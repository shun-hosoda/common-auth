## 実装方針

### アーキテクチャ

**ミドルウェアスタック**（実行順序重要）

```
Request
   ↓
1. SecurityHeadersMiddleware  ← 全レスポンスにヘッダ付与
   ↓
2. JWTAuthMiddleware         ← JWT検証、request.state.userに格納
   ↓
3. TenantMiddleware          ← SET LOCAL app.current_tenant_id
   ↓
Endpoint Handler
   ├─ Depends(get_current_user)  ← request.state.userから取得
   └─ Depends(get_tenant_id)     ← request.state.tenant_idから取得
   ↓
Response
```

**setup_auth()での一括登録**

```python
from common_auth import AuthConfig, setup_auth

app = FastAPI()
config = AuthConfig.from_env()
setup_auth(app, config)  # ミドルウェア+ルーター一括登録
```

### コーディング規約

| 項目 | 規約 |
|---|---|
| 関数サイズ | 30行以内（複雑なロジックは分割） |
| 型ヒント | すべての関数にType Hints必須 |
| エラーハンドリング | カスタム例外を定義、Middlewareでcatch |
| ロギング | structlog使用、機密情報をログに出さない |
| 命名規則 | PEP8準拠、snake_case |
| Docstring | Google Style、公開APIには必須 |

## 議論のポイント

### 論点1: JWKSキャッシュのTTL設定

**Sec**: 「24時間は長すぎないか？鍵ローテーション時に最大24時間古い鍵で検証され続ける」

**Arch**: 「Keycloakのデフォルト設定では、新旧両方の鍵が並行して有効（Grace Period）。未知のkidを検出した時点で即座にリフレッシュするため、実質的な遅延は1リクエスト分のみ。24時間は妥当」

**結論**: 24時間TTL採用。環境変数`JWKS_CACHE_TTL`で調整可能にする。

### 論点2: Lazy Syncのデフォルト動作

**DB**: 「Lazy Syncをデフォルト有効にすると、業務DBを持たないプロジェクトでエラーになる」

**PM**: 「backend-sdk-details.mdでオプトインと記載済み。デフォルト無効で、`ENABLE_USER_SYNC=true`で有効化」

**結論**: デフォルト無効、環境変数で有効化。

### 論点3: テストでのKeycloak環境

**Eng**: 「ユニットテストではモック、E2EではDocker Compose起動。統合テストは？」

**Arch**: 「統合テストもモックでOK。JWKS取得部分だけ`httpx_mock`で差し替え。E2Eで実Keycloak環境を検証すれば十分」

**結論**: ユニット+統合はモック、E2Eのみ実Keycloak。
