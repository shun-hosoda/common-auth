# レビューチェックリスト

レビューボードが議論の中で確認すべき項目。
全項目をチェックリストとして順に埋めるのではなく、議論の材料として使う。

## 設計フェーズ（Gate 1）

### PM観点
- [ ] PRDの機能要件がすべてAPI仕様に反映されている
- [ ] ユーザーストーリーが具体的で測定可能
- [ ] MVPスコープが適切（過大でも過小でもない）

### Architect観点
- [ ] コンポーネント間の依存関係が一方向
- [ ] DBスキーマがAPI仕様のデータモデルと整合
- [ ] 非機能要件が設計に数値目標付きで反映されている
- [ ] ADRで主要な技術選定の根拠が記録されている
- [ ] **フロントエンドが呼ぶAPIパスとバックエンドのルーターprefix+パスが完全一致している**
- [ ] **Vite proxy設定が全APIプレフィックス（`/api`, `/auth` 等）をカバーしている**

## 実装フェーズ（Gate 2）

### Engineer観点
- [ ] 設計書と実装が一致している
- [ ] エラーハンドリングが網羅的（ハッピーパス以外も）
- [ ] テストカバレッジが主要パスを網羅
- [ ] DRY原則に従い重複コードがない

### デグレ防止観点（⚠️ 必須確認）

以下は過去にデグレを引き起こした項目。PRで変更されたファイルとの関連を必ず確認する。

#### A. 外部API呼び出しのペイロード整合性
- [ ] Keycloak Admin REST API を呼ぶ PUT/POST は、**完全なリソース表現**（UserRepresentation, GroupRepresentation 等）を送信しているか？（部分オブジェクトは 400 Bad Request になる）
- [ ] テストの PUT ボディ assertion が、必須フィールド（`id`, `username`, `email` 等）の存在を検証しているか？
- [ ] モック応答が実際のKeycloak APIレスポンス形式と一致しているか？（`attributes` が null/未設定の場合を含む）

#### B. ミドルウェア/共有パス上の例外スコープ
- [ ] ミドルウェアの try-except が `call_next(request)` を含んでいないか？（ハンドラーの例外がミドルウェアのエラーとして誤キャッチされる）
- [ ] 認証ミドルウェアの例外処理が、認証固有の例外（`ExpiredSignatureError`, `TokenInvalidError`）**のみ**をキャッチしているか？

#### C. フロントエンド↔バックエンド パス整合性
- [ ] フロントエンドの API クライアントが呼ぶ URL パスと、バックエンドの `include_router(prefix=...)` + エンドポイントパスの組み合わせが一致しているか？
- [ ] Vite proxy 設定が全てのバックエンドプレフィックスをカバーしているか？
- [ ] **`baseOverride` やハードコードされた base URL** がルーター prefix と一致しているか？

#### D. 環境変数・シークレットの整合性
- [ ] `.env` のシークレット値が、実際の外部サービス（Keycloak 等）の現在の設定と一致しているか？
- [ ] `docker compose down -v` 後の再起動でシークレットが変わる可能性を考慮しているか？
- [ ] `.env.example` が現在の実値と同期しているか？

### Security観点
- [ ] 全ユーザー入力がサニタイズされている
- [ ] SQLクエリがパラメータ化されている
- [ ] 認証・認可が適切に実装されている
- [ ] 機密情報がログ・レスポンスに露出していない

### DB観点
- [ ] インデックスがクエリパターンに対応している
- [ ] N+1クエリが発生しない設計
- [ ] トランザクション境界が適切
- [ ] マイグレーションにロールバック手順がある

## リリースフェーズ（Gate 3）

- [ ] 全テストがCI上でパスしている
- [ ] 環境変数・シークレットが適切に管理されている
- [ ] パフォーマンス基準（PRDの非機能要件）を達成
- [ ] ロールバック・障害時の復旧手順が明確
- [ ] **スモークテスト (`scripts/smoke-test.ps1`) が全エンドポイントでパスしている**
