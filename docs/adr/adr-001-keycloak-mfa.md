# ADR-001: Keycloak MFA 方式の選定

## ステータス
承認済み（v1.1 — /fix 反映済み）

## コンテキスト
複数顧客への配布を前提とした認証基盤において、MFA方式を選定する必要がある。

## 決定

1. **メールOTP**: Keycloak v24標準の Email OTP Authenticator を使用する
2. **TOTP**: Keycloak標準の OTP Authenticator + Required Action を使用する
3. **DB**: PostgreSQL（H2は開発モード限定、本番不可のため）
4. **Keycloak バージョン**: v24以降（Email OTP標準搭載バージョン）

### realm-export ファイル分離方針（追記）

`realm-export.json`（メールOTP用）と `realm-export-totp.json`（TOTP用）を別ファイルとして管理する。

**理由:** 顧客A・顧客Bは認証フローが根本的に異なるため、1ファイルへの統合は設定の複雑性を増す。
共通設定の二重管理は環境変数（`.env`）による注入で吸収し、顧客切り替えはファイル選択で行う。

**トレードオフ:** 共通設定（SMTP等）の変更時は両ファイルを更新する必要がある。
この運用コストを許容した上で「顧客ごとの設定明確性」を優先する。

### シークレット管理方針（追記）

- `.env.example` はプレースホルダーのみ。実値は `.env`（gitignore対象）に記載
- リポジトリへのシークレットコミットを禁止する

## 却下した選択肢

| 選択肢 | 却下理由 |
|--------|---------|
| SMS OTP | 外部キャリアAPI依存、コスト・可用性リスク |
| WebAuthn | 顧客端末要件が高い、MVP対象外 |
| カスタムSPI実装 | 保守コスト大、標準機能で要件を満たせる |
| realm-export 1ファイル統合 | 設定の複雑性増大、顧客別設定の視認性低下 |

## 影響
- `realm-export.json` にフロー定義を集約することで、顧客配布・環境再現が容易になる
- Keycloak バージョンアップ時は Email OTP の設定キー変更に注意が必要
- `.env` の gitignore 管理を徹底しないとシークレット漏洩リスクがある
