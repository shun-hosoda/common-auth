# ADR-004: Backend JWT検証ライブラリの選定（PyJWT）

## ステータス

承認

## コンテキスト

Backend SDK（Python/FastAPI）でKeycloakが発行するJWTのRS256署名を検証するための
ライブラリを選定する必要がある。

## 決定

**PyJWT + cryptography を採用する。**

## 選択肢

### 選択肢A: PyJWT + cryptography（採用）
- **メリット**:
  - Python JWT処理のデファクトスタンダード
  - アクティブにメンテナンスされている
  - RS256/RS384/RS512、ES256等の非対称鍵アルゴリズムに対応
  - `cryptography`バックエンドで高性能
  - APIがシンプルで学習コスト低
- **デメリット**:
  - JWKS自動取得機能はなく、自前でHTTPフェッチ+キャッシュの実装が必要

### 選択肢B: python-jose
- **メリット**: JWKSサポートが組み込み
- **デメリット**: メンテナンス頻度が低下傾向、依存関係の選択肢が複雑

### 選択肢C: Authlib
- **メリット**: OIDC/OAuthクライアント機能も含む包括的ライブラリ
- **デメリット**: 認証モジュールにはオーバースペック、学習コスト高

## 結果

- PyJWTのシンプルなAPIでJWT検証ロジックを実装
- JWKS取得・キャッシュは自前実装（`httpx` + TTLキャッシュ）
  - TTL: 24時間
  - 未知のkidを検出した場合は即時リフレッシュ
- `cryptography`パッケージでRS256署名検証のパフォーマンスを確保

### JWKS取得失敗時のGraceful Degradation戦略

Keycloak障害時の挙動を定義する:

| 状況 | 挙動 | HTTPステータス |
|---|---|---|
| JWKS初回取得失敗（起動時） | エラーログ出力、起動継続、次回リクエスト時に再試行 | - |
| JWKSキャッシュ有効（24h以内） | キャッシュで検証継続 | 200（正常） |
| JWKSキャッシュ期限切れ + 再取得失敗 | 503エラーを返す | 503 |
| 未知のkid検出 + 即時リフレッシュ失敗 | 503エラーを返す | 503 |
| トークン検証成功後の後続リクエスト | 既存キャッシュで検証継続（新規取得なし） | 200 |

**設計判断**:
- 可用性優先: JWKSキャッシュが有効な間は認証を継続
- セキュリティ担保: キャッシュ期限切れ時は認証を停止（古い鍵での検証リスク回避）
- 運用考慮: `/auth/health`エンドポイントでJWKS取得状態を監視可能
