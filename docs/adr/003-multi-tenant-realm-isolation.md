# ADR-003: マルチテナント分離方式（Realm分離 + tenant_id抽出戦略）

## ステータス

承認

## コンテキスト

共通認証モジュールは顧客ごとの独立デプロイと、1インスタンスでの複数テナント運用の
両方に対応する必要がある。テナントごとのデータ分離と、業務DBでのテナント識別方法を
決定する必要がある。

## 決定

1. **Keycloak Realm per tenant** によるテナント分離を採用する
2. **tenant_id抽出**はデフォルトでJWT `iss`クレームからRealm名を導出する
3. 設定により以下も選択可能とする：
   - カスタムJWTクレーム（Protocol Mapper設定時）
   - 環境変数での固定値（単一テナント構成時）

## 選択肢

### テナント分離方式

#### 選択肢A: Realm per tenant（採用）
- **メリット**:
  - ユーザー、クライアント、ロールが完全に分離
  - Realm設定のJSONエクスポート/インポートでポータブル
  - テナント間の設定差異（MFA要否、パスワードポリシー等）に対応可能
- **デメリット**:
  - テナント数が数百を超えるとKeycloakのパフォーマンスに影響
  - Realm追加のたびにKeycloak設定が必要

#### 選択肢B: 単一Realm + グループ/属性
- **メリット**: Realm管理のオーバーヘッドなし
- **デメリット**: テナント間の設定差異に対応しにくい、分離が論理的のみ

### tenant_id抽出方式

#### 選択肢A: `iss`クレームからRealm名抽出（デフォルト採用）
- **メリット**: Keycloakの追加設定不要、標準クレームのみで動作
- **デメリット**: Realm名 = tenant_idの制約

#### 選択肢B: カスタムクレーム（設定で選択可能）
- **メリット**: tenant_idを自由に定義可能
- **デメリット**: Keycloak Protocol Mapper設定が必要

#### 選択肢C: 環境変数固定値（設定で選択可能）
- **メリット**: 単一テナントで最もシンプル
- **デメリット**: マルチテナント不可

## 結果

- 顧客ごとの独立デプロイ（単一Realm）と複数テナント運用（複数Realm）の両方に対応
- デフォルト設定では追加のKeycloak設定なしでテナント識別が動作
- テナント数が100未満の構成を前提とする（それ以上は要検証）
- 業務DBの全テーブルに`tenant_id`カラムを定義し、ミドルウェアで自動フィルタリング
