# ADR-001: IdPとしてKeycloakを採用

## ステータス

承認

## コンテキスト

共通認証モジュールにはOIDC/OAuth 2.0準拠のIdentity Providerが必要である。
「Don't Roll Your Own Auth」の原則に従い、認証ロジックの自前実装を避け、
実績あるソリューションを採用する。また、特定クラウドに依存しない
ポータブルな構成が求められる。

## 決定

**Keycloak（Docker）をIdPとして採用する。**

## 選択肢

### 選択肢A: Keycloak（Docker）
- **メリット**:
  - OSS（Apache 2.0）でライセンスコスト不要
  - OIDC/OAuth 2.0/SAMLの全プロトコルに対応
  - ユーザー管理、MFA、パスワードリセット、セルフ登録を標準搭載
  - Dockerコンテナで任意の環境にデプロイ可能（ポータブル）
  - Realm機能によるマルチテナント分離
  - Admin Console/APIでの管理機能が充実
  - Red Hat支援の活発なコミュニティ
- **デメリット**:
  - 自前でのインフラ運用が必要（可用性、バックアップ）
  - JVMベースで初期メモリ消費が大きい（512MB〜）
  - バージョンアップ時のマイグレーション作業

### 選択肢B: AWS Cognito
- **メリット**: マネージドで運用不要、AWS統合が容易
- **デメリット**: AWS固有、ポータビリティを満たさない、カスタマイズ制約

### 選択肢C: Auth0
- **メリット**: マネージドでDXが良い、豊富なSDK
- **デメリット**: 商用ライセンス、コストが高い、クラウド依存

### 選択肢D: 自前実装
- **メリット**: 完全な制御
- **デメリット**: 「Don't Roll Your Own Auth」に反する、セキュリティリスク、開発コスト大

## 結果

- Keycloakにより認証関連の機能（ユーザー管理、MFA、パスワードリセット等）を自前実装せずに済む
- Docker Composeで任意の顧客環境にデプロイ可能
- 将来的にクラウドマネージドサービスへの移行も可能（OIDC準拠のため）
- インフラ運用コスト（監視、バックアップ、バージョンアップ）は許容する
