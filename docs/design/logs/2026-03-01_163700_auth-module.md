# 設計会議記録 — 共通認証・ユーザー管理モジュール

## 日時
2026-03-01

## 参加者
PM, Architect, DB Specialist, Security Specialist, Senior Engineer
+ ドメインペルソナ: 認証基盤テックリード

## 要件サマリー

- 複数プロジェクトで再利用可能なポータブル認証プラットフォーム
- Keycloak（Docker）をIdPとして採用し、認証ロジックの自前実装を回避
- ステートレス & スケーラブルなJWTベースのアーキテクチャ
- Frontend SDK（React Hooks）+ Backend SDK（FastAPI Middleware）として配布
- マルチテナント対応（Keycloak Realm分離）
- クラウド非依存（Docker Composeベース）

## 設計決定

### アーキテクチャ

- **Auth Stack**（Keycloak + PostgreSQL）と**SDK Layer**（Frontend/Backend）を完全分離
- Auth StackはDocker Composeで顧客環境に独立デプロイ
- SDK Layerはnpm / PyPIパッケージとして配布
- バックエンドはステートレス（セッション不使用、JWT検証のみ）

### 認証フロー

- OIDC Authorization Code Flow + PKCE（OAuth 2.1推奨）
- RS256署名、JWKS公開鍵キャッシュ（TTL 24h、未知kid即時refresh）
- Implicit Flow / Password Grant は禁止

### API設計（Backend SDK提供のオプショナルルーター）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| GET | `/auth/me` | 現在のユーザー情報取得 | 必要 |
| GET | `/auth/health` | Auth Stack死活監視 | 不要 |
| POST | `/auth/logout` | トークン失効（Keycloak連携） | 必要 |

### DB設計

- **認証データ**: Keycloak内部DB（パスワード、MFA等）→業務DBに保持しない
- **業務DB**: `tenants`テーブル + `user_profiles`テーブル（推奨パターン）
- `user_profiles.id` = Keycloakの`sub`クレーム（不変UUID）
- Lazy Sync（JWT検証時にupsert）はオプトイン
- RLSポリシーテンプレートを提供

### マルチテナント設計

- Keycloak Realm per tenant
- tenant_id取得のデフォルト: JWT `iss`クレームからRealm名を抽出
- 設定で切替可能: カスタムクレーム / 環境変数固定値

### セキュリティ設計

- SecurityHeadersミドルウェア（HSTS, CSP, X-Frame-Options等）
- Rate Limitingミドルウェア（Phase 2）
- CORS: Keycloak + Frontend originのみ許可
- アクセストークンはメモリ保持（XSS対策）

### 実装方針

- **Backend SDK**: Python / FastAPI / PyJWT + cryptography / httpx
- **Frontend SDK**: React 18+ / TypeScript / oidc-client-ts
- `pip install common-auth` / `npm install @common-auth/react` で導入完了
- 環境変数3つ（KEYCLOAK_URL, KEYCLOAK_REALM, KEYCLOAK_CLIENT_ID）で動作

## 議論のポイント

- **tenant_id抽出方式**: DB Specialistがカスタムクレーム（Protocol Mapper）を推したが、PMが「追加設定なしで動くべき」と主張。`iss`クレームからRealm名抽出をデフォルトとし、config切替で対応に決着
- **user_profiles同期方式**: EngがKeycloakイベントリスナーを提案、Archが複雑度を理由にMVPではLazy Sync（JWT初回検証時にupsert）に限定を提案、PM同意でLazy Sync（オプトイン）に決定
- **CSP設定**: Secがstrictデフォルトを主張、Engが業務アプリで上書きが必要と指摘。strictデフォルト + configでカスタマイズ可能に決着
- **keycloak-js vs oidc-client-ts**: Archがポータビリティの観点からoidc-client-tsを推奨。将来のIdP移行時にSDK差替が不要になる点が決め手

## 起票したADR

- ADR-001: IdPとしてKeycloakを採用（ポータビリティ + Don't Roll Your Own Auth）
- ADR-002: OIDC Authorization Code Flow + PKCEの採用（OAuth 2.1推奨フロー）
- ADR-003: マルチテナント分離方式（Realm分離 + tenant_id抽出戦略）
- ADR-004: Backend JWT検証ライブラリの選定（PyJWT + cryptography）
- ADR-005: Frontend OIDCライブラリの選定（oidc-client-ts）
