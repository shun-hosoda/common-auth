### Backend SDK設計

#### 1. Rate Limiting Middleware

**ファイル**: `packages/backend-sdk/src/common_auth/middleware/rate_limit.py`

**実装方針**:
- アルゴリズム: Token Bucket方式
- ストレージ: メモリキャッシュ（`cachetools.TTLCache`）
- 抽象化: `RateLimitStore` インターフェース定義（将来のRedis対応を視野）

**クラス設計**:
```python
from abc import ABC, abstractmethod
from cachetools import TTLCache
from starlette.middleware.base import BaseHTTPMiddleware

class RateLimitStore(ABC):
    """Rate limiting状態を管理するストレージの抽象クラス。"""
    
    @abstractmethod
    def check_and_increment(self, key: str, limit: int, window: int) -> tuple[bool, int]:
        """
        レート制限をチェックし、カウンターを増分する。
        
        Args:
            key: クライアント識別子（通常はIP）
            limit: 制限値（リクエスト数）
            window: 時間窓（秒）
        
        Returns:
            (許可/拒否, 残りリクエスト数)
        """
        pass

class InMemoryRateLimitStore(RateLimitStore):
    """メモリベースのRate Limitingストレージ（単一インスタンス用）。"""
    
    def __init__(self, maxsize: int = 10000):
        self._cache: TTLCache = TTLCache(maxsize=maxsize, ttl=60)
        self._lock = threading.Lock()
    
    def check_and_increment(self, key: str, limit: int, window: int) -> tuple[bool, int]:
        with self._lock:
            current = self._cache.get(key, 0)
            if current >= limit:
                return (False, 0)
            self._cache[key] = current + 1
            remaining = limit - (current + 1)
            return (True, remaining)

class RateLimitMiddleware(BaseHTTPMiddleware):
    """
    Token Bucket方式のRate Limiting Middleware。
    
    機能:
    - IP単位でリクエスト数制限
    - エンドポイント別に異なる制限値を設定可能
    - X-RateLimit-* ヘッダーを返却
    - 429 Too Many Requests + Retry-After
    """
    
    def __init__(
        self,
        app,
        config: AuthConfig,
        store: RateLimitStore | None = None
    ):
        super().__init__(app)
        self.config = config
        self.store = store or InMemoryRateLimitStore()
        self.limits = self._parse_limits(config)
    
    def _parse_limits(self, config: AuthConfig) -> dict[str, int]:
        """エンドポイント別のレート制限値を解析。"""
        return {
            "/auth/login": config.rate_limit_login_requests,
            "default": config.rate_limit_default_requests
        }
    
    async def dispatch(self, request: Request, call_next):
        if not self.config.rate_limit_enabled:
            return await call_next(request)
        
        # クライアントIP取得
        client_ip = self._get_client_ip(request)
        
        # エンドポイント別制限値取得
        path = request.url.path
        limit = self.limits.get(path, self.limits["default"])
        
        # Rate Limit チェック
        key = f"ratelimit:{client_ip}:{path}"
        allowed, remaining = self.store.check_and_increment(key, limit, 60)
        
        if not allowed:
            return JSONResponse(
                status_code=429,
                content={"error": "too_many_requests", "message": "Rate limit exceeded"},
                headers={
                    "Retry-After": "60",
                    "X-RateLimit-Limit": str(limit),
                    "X-RateLimit-Remaining": "0"
                }
            )
        
        # 正常処理
        response = await call_next(request)
        response.headers["X-RateLimit-Limit"] = str(limit)
        response.headers["X-RateLimit-Remaining"] = str(remaining)
        return response
    
    def _get_client_ip(self, request: Request) -> str:
        """
        実クライアントIPを取得。X-Forwarded-Forを考慮。
        
        Args:
            request: FastAPI Request
        
        Returns:
            クライアントIP文字列
        """
        # X-Forwarded-For検証
        forwarded = request.headers.get("X-Forwarded-For")
        if forwarded and self._is_trusted_proxy(request.client.host):
            # 最初のIPを返す（クライアント側）
            return forwarded.split(",")[0].strip()
        return request.client.host
    
    def _is_trusted_proxy(self, ip: str) -> bool:
        """プロキシが信頼リストに含まれるか確認。"""
        # CIDR範囲チェック（実装省略）
        return True  # 暫定実装
```

### セキュリティ設計

#### 1. MFA (TOTP)

- **実装**: Keycloak標準のTOTP（RFC 6238準拠）
- **設定**: Required Actionとして`CONFIGURE_TOTP`を有効化
- **ユーザー体験**: 初回ログイン後にQRコード表示、Google Authenticator等でスキャン

#### 2. パスワードリセット

- **フロー**:
  1. ユーザーが「パスワードを忘れた」をクリック
  2. Keycloakがリセットリンク付きメールを送信
  3. ユーザーがリンクをクリック、新しいパスワード入力
  4. Keycloakがパスワードを更新

- **セキュリティ対策**:
  - リセットトークンの有効期限: 12時間（Keycloak設定可能）
  - トークンは1回のみ使用可能
  - リセット完了後、全セッション無効化

#### 3. ユーザーセルフ登録

- **フロー**:
  1. ユーザーが登録フォーム入力
  2. Keycloakが確認メール送信
  3. ユーザーがリンクをクリックしてメール確認
  4. アカウント有効化

- **セキュリティ対策**:
  - メール確認必須（`verifyEmail: true`）
  - パスワードポリシー: 最小8文字、大小英字・数字を含む
  - （オプション）reCAPTCHA v3でボット対策

#### 4. Rate Limiting

- **対策するAttack**:
  - ブルートフォース攻撃（パスワード総当たり）
  - クレデンシャルスタッフィング（漏洩パスワード辞書攻撃）
  - API abuse（過剰なリクエスト）

- **制限値**:
  - `/auth/login`: 5 req/min（厳しく）
  - その他API: 60 req/min（緩やか）
  - 制限超過時: 429 Too Many Requests + Retry-After: 60

- **X-Forwarded-For対策**:
  - 信頼プロキシのCIDR範囲を設定可能
  - 範囲外のプロキシからの`X-Forwarded-For`は無視

## 議論のポイント

### 論点1: Rate Limitingのストレージ選択

**DB Specialist**: 複数のBackend SDKインスタンス（水平スケール）で動作させる場合、メモリキャッシュでは状態が共有されず、レート制限が効かなくなる。Redisのような共有ストレージが必要ではないか？

**Architect**: 指摘は正しい。ただし、Phase 2aではシンプルさとポータビリティを優先してメモリ実装を選択する。将来的なRedis対応を容易にするため、`RateLimitStore`という抽象クラスを定義し、実装を差し替え可能にする。

**Engineer**: 同意。`InMemoryRateLimitStore`と`RedisRateLimitStore`を同じインターフェースで実装すれば、切り替えは容易。Phase 3でRedis対応を実装する。

**決定**: 
- Phase 2a: `InMemoryRateLimitStore`（単一インスタンス前提）
- `RateLimitStore`抽象クラスを定義
- Phase 3: `RedisRateLimitStore`実装

### 論点2: Frontend SDKの実装優先度

**Product Manager**: Backend SDKだけでも価値はあるが、開発者体験としてFrontend SDKもセットで提供したい。Phase 2で両方実装すべきか？

**Senior Engineer**: Phase 1でBackend SDKのパターンが確立しているので、同様のアプローチでFrontend SDKも実装できる。実装工数の内訳は、React Hooks実装（2日）、単体テスト作成（1-2日）、ドキュメント作成（1日）で、合計**1週間程度**を見込むべき。

**Architect**: Phase 2を2段階に分けるのはどうか？Phase 2aでBackend拡張（Rate Limiting + Keycloak設定）、Phase 2bでFrontend SDK実装。

**決定**: Phase 2を2段階に分割
- Phase 2a（Must）: Backend SDK + Auth Stack
- Phase 2b（Should）: Frontend SDK + Example App

### 論点3: SMTP認証情報の管理

**Security Specialist**: `.env`ファイルにSMTPパスワードを平文保存するのはセキュリティリスク。本番環境では別の方法を推奨すべき。

**Architect**: 同意。ただし、開発環境での利便性も重要。以下の方針を提案:
1. `.env.example`にはプレースホルダーのみ記載
2. READMEにベストプラクティスを明記:
   - 本番: AWS Secrets Manager, Azure Key Vault, HashiCorp Vault
   - Docker: Docker Secrets機能
   - Kubernetes: Sealed Secrets

**Engineer**: Keycloakの環境変数置換機能（`#{VAR_NAME}`）を使えば、シークレット管理ツールからの値注入も容易。

**決定**:
- `.env`は開発環境用として許容
- `.env.example`にセキュリティ警告を記載
- READMEに本番環境でのシークレット管理ベストプラクティスを詳述

### 論点4: MFAの強制ポリシー

**Security Specialist**: MFAはオプションではなく、特定の役割（管理者等）に対して強制すべきではないか？

**Product Manager**: Phase 2の要件では「MFA対応」であり、強制ではない。ユーザーが任意で有効化できる状態を目指す。MFA強制ポリシーはPhase 3で検討。

**Architect**: Keycloak設定で`CONFIGURE_TOTP`をRequired Actionの`defaultAction: false`にすれば、任意設定になる。将来的に特定グループに対して`defaultAction: true`にすることも可能。

**決定**: Phase 2はMFA任意設定。Phase 3で役割ベースのMFA強制ポリシーを検討。

## 起票すべきADR

### ADR-007: Rate Limiting方式の選定

**決定**: Token Bucket方式を採用

**理由**:
- 実装がシンプル
- メモリ効率が良い（Fixed Windowより少ない状態）
- バースト対応（短時間の急激なリクエストを吸収）

**代替案**:
- Sliding Window: より正確だが実装複雑
- Fixed Window: シンプルだがバースト脆弱性あり

**ストレージ**:
- Phase 2a: メモリ（`cachetools.TTLCache`）
- Phase 3: Redis対応（水平スケール対応）

### ADR-008: Frontend SDK技術選定

**決定**: `oidc-client-ts` + React Hooks wrapper

**理由**:
- ADR-005で既に`oidc-client-ts`を選定済み
- OIDC標準準拠、IdP非依存
- React Hooksでラップすることで使いやすいAPIを提供

**代替案**:
- `react-oidc-context`: `oidc-client-ts`のReact wrapper（公式）
  - 検討したが、カスタマイズ性を優先して独自wrapper実装

### ADR-009: Email送信方式

**決定**: Keycloak内蔵SMTP機能を使用

**理由**:
- ポータビリティ優先（外部API依存を避ける）
- 設定が`.env`で完結
- Dockerコンテナのみで動作

**代替案**:
- 外部Email API（SendGrid, AWS SES等）
  - クラウド依存が発生するため不採用

**トレードオフ**:
- Keycloak内蔵SMTPは送信レート制限が厳しい場合がある
- 大規模運用時は外部APIへの切り替えも検討
